# 代码方案综合评估（Android 混合开发增强版）

## 触发条件

用户提问包含以下关键词且意图为**评估代码方案/设计/思路**时触发：

- 怎么样
- 你觉得咋样
- 代码如何
- 评估一下
- 分析一下
- 可行吗

> 注意：单纯询问“如何实现”且无评估意图时不触发；若用户仅给出模糊描述，优先引导其补充关键细节后再评估。

***

## 评估维度

### 1. 架构评估

评估代码方案的整体架构设计是否合理。

**检查项：**

- 模块划分是否清晰
- 层次结构是否合理
- 依赖关系是否正确
- 是否符合项目现有架构风格
- 扩展性如何
- **Android 专项：** 导航架构是否统一（单 Activity / 多 Fragment / Compose Navigation）
- **Android 专项：** Compose 与 Fragment/View 的职责划分是否清晰，避免架构混杂

### 2. 逻辑评估

评估实现思路和业务逻辑的正确性。

**检查项：**

- 业务逻辑是否完整
- 边界条件是否考虑
- 异常情况是否处理
- 流程是否清晰易懂
- 是否存在逻辑漏洞
- **状态同步专项：** 全局配置/状态的变更能否实时通知所有依赖方
- **状态同步专项：** 是否存在多个数据源副本导致 UI 显示不一致
- **状态同步专项：** 切换/选择操作后，所有相关 UI 是否基于同一数据源渲染
- **状态同步专项：** 导出、提交等异步操作是否取用最新的配置值，而非组件创建时的快照

**业务逻辑深度检查（必须执行）：**

- **数据源一致性检查：**
  - 是否存在新旧数据源并存的情况（如：ACache缓存 vs Room数据库）
  - 用户配置保存的位置与读取的位置是否一致
  - 是否存在"保存到数据库，但从缓存读取"的不一致情况
  - 多个模块访问同一数据时，是否使用相同的数据源

- **新旧架构兼容性检查：**
  - 是否存在新旧两套实现并存的情况
  - 新功能是否正确使用了新的架构（如：新的Repository、ViewModel）
  - 旧代码调用新功能时，数据流是否正确
  - 迁移期间是否有数据丢失或显示错误的风险

- **用户操作实际效果验证：**
  - 用户在界面上的操作（选择、切换、配置）是否真正生效
  - 操作结果是否能被后续流程正确读取
  - 是否存在"界面显示成功，但实际未生效"的情况
  - 端到端测试：用户操作 → 数据保存 → 数据读取 → UI显示，整个链路是否通畅

- **配置管理专项检查：**
  - 配置项是否有默认值机制
  - 用户修改配置后，是否影响所有依赖该配置的功能
  - 配置的读取时机是否正确（实时读取 vs 缓存快照）
  - 配置变更是否有通知机制，确保所有使用方更新

- **跨模块数据流检查：**
  - 数据在模块间传递时是否保持一致性
  - 是否存在绕过正常数据流直接访问底层数据的情况
  - 异步操作（网络请求、文件操作）使用的配置是否是最新的

### 3. 可行性评估

评估方案在技术上是否可行。

**检查项：**

- 技术选型是否合适
- 是否依赖不可用的资源
- 是否存在技术瓶颈
- 时间成本是否可接受
- 是否有更好的替代方案
- **Android 专项：** Compose 与旧版 View 三方库的兼容性是否经过验证

### 4. 实现难度评估

评估方案的开发复杂度。

**检查项：**

- 代码量预估
- 依赖项复杂度
- 是否需要引入新框架
- 学习成本
- 开发周期预估
- **Android 专项：** 从 View 到 Compose 的渐进式迁移成本是否可控

### 5. 调试难度评估

评估方案的问题排查难度。

**检查项：**

- 日志覆盖是否充分
- 错误信息是否明确
- 是否易于单元测试
- 问题定位难度
- 监控埋点是否完善
- **Android 专项：** 混合布局检查器（Layout Inspector）是否能同时显示 View 树和 Compose 节点
- **Android 专项：** 重组崩溃的堆栈能否快速定位到业务代码

### 6. 可维护性评估

评估方案的长期维护成本。

**检查项：**

- 代码可读性
- 注释和文档是否完善
- 是否易于理解和修改
- 是否存在过度设计
- 是否符合团队编码规范

### 7. 性能评估

评估方案的性能表现。

**检查项：**

- 时间复杂度分析
- 空间复杂度分析
- 是否存在性能瓶颈
- 资源消耗预估
- 并发安全性
- **Android 专项：** 列表混合滑动（RecyclerView 嵌套 Compose / LazyColumn 嵌套 View）的帧率稳定性
- **Android 专项：** 重组是否会在 View 宿主触发全量 measure/layout
- **Android 专项：** Proguard/R8 规则是否影响 Compose 运行时

### 8. Android Compose/View 混合开发适配评估（特化维度）

专门针对同时使用 Jetpack Compose 和传统 View 体系的方案。

**检查项：**

- 互操作 API 使用是否合理（`AndroidView`、`ComposeView`、`AndroidViewBinding`）
- 生命周期与 `SavedState` 在双系统间是否一致
- 主题/配置变化（暗黑模式、语言等）的同步机制
- 状态观察冲突预防（`LiveData` ↔ `StateFlow` ↔ Compose `State`）
- 手势分发与焦点管理在混合层级中是否正常
- 是否存在常见的性能反模式（重复重组、冗余布局嵌套）
- 渐进式迁移路径是否通畅（能否平滑切换为纯 Compose 或纯 View）

### 9. 状态同步与 UI 一致性评估（业务逻辑特化维度）

针对“一处配置变更，多处 UI 应联动更新”的场景，专门检查方案是否可靠。

**检查项：**

- 全局配置（如上传配置选择）是否保存在单一数据源（Repository / DataStore / 内存单例）
- 所有 UI 组件是否通过同一个可观察流（`StateFlow` / `LiveData` / Compose `State`）读取配置
- 切换/更改操作是否直接修改数据源，并保证所有观察者收到更新
- 是否存在直接读取静态值或 Application 缓存，绕过观察机制的情况
- 在混合 UI（View 和 Compose）中，观察者是否绑定正确的生命周期（避免漏更新或内存泄漏）
- 异步操作（文件导出、网络请求）使用的配置值，是否在操作执行时实时获取，而非捕获旧引用

### 10. 潜在风险评估

识别方案可能存在的问题和风险。

**检查项：**

- 安全风险
- 兼容性风险（版本、设备、API Level）
- 数据一致性风险
- 依赖稳定性风险
- 回滚方案是否完备
- **Android 专项：** Compose 与 Compose Compiler 的 Kotlin 版本不匹配风险
- **Android 专项：** 第三方 View 库在 `ComposeView` 内嵌时可能不正常工作
- **Android 专项：** 不混淆规则缺失导致 Compose 运行时崩溃
- **状态同步风险：** 配置变更未通知到某些 UI 组件，导致显示与所选不一致（高频坑）

## 执行流程

```
1. 接收用户评估请求
      ↓
2. 拆解用户需求与代码思路（若信息不足，暂停并提问）
      ↓
3. 架构评估
      ↓
4. 逻辑评估（含状态同步基础检查）
      ↓
5. 可行性评估
      ↓
6. 实现难度评估
      ↓
7. 调试难度评估
      ↓
8. 可维护性评估
      ↓
9. 性能评估
      ↓
10. Android Compose/View 混合开发适配评估（如涉及相关技术栈）
      ↓
11. 状态同步与 UI 一致性评估（如涉及跨组件状态共享）
      ↓
12. 潜在风险评估
      ↓
13. 综合结论
      ↓
14. 提供优化方案（必要时给出多方案对比）
      ↓
15. 输出完整评估报告
```

## 注意事项

1. 评估必须客观公正，不偏袒任何方案。
2. 每个维度都要给出具体分析，不能只给评分。
3. 发现问题必须提供对应的解决方案。
4. 综合结论要明确，不能模棱两可。
5. 优化方案要切实可行，不能只是理论上的。
6. 若项目明确为 Android 且涉及 UI 部分，务必启用第 8、第 9 两个特化维度检查。
7. 当用户描述缺乏必要上下文（如无具体代码、架构不清）时，应先提出 1-2 个澄清问题，而非直接空泛评估。
8. 多方案对比时，应保持维度一致，并最终给出推荐。
9. **业务逻辑深度检查是强制项**，不能因为代码结构看似正确就跳过。必须验证：
   - 数据源是否一致（保存和读取是否使用同一数据源）
   - 新旧架构是否兼容（是否存在数据流断裂）
   - 用户操作是否真正生效（端到端验证）
10. 特别警惕"代码质量高但业务逻辑错误"的情况，这类问题往往隐藏更深，影响更严重。
