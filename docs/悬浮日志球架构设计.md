# 悬浮日志球架构设计

第二份偏工程架构，包含事件模型、采集层、事件中心、traceId、悬浮球实现、UI、性能和风险点

## 设计目标

构建一个应用内实时调试控制台，通过页面右下角悬浮球进入。系统需要统一采集网络请求、应用错误、书源/RSS 源规则执行、书源校验等事件，并以低侵入、可扩展的方式展示给用户。

设计重点：

- 采集和展示解耦。
- 日志事件结构化。
- 实时展示，不依赖 UI 轮询。
- 默认低开销，避免影响阅读和网络性能。
- 对请求体、响应体、Cookie、Token 等敏感内容做保护。
- 能复用现有 `AppLog`、`UrlRecordInterceptor`、`Debug`、书源调试和校验能力。

## 总体分层

建议分为四层：

```text
采集层
  ↓
调试事件中心
  ↓
调试数据仓库
  ↓
悬浮入口与调试 UI
```

## 采集层

采集层负责把不同模块里的运行信息转换成统一的调试事件。

主要采集点：

- 网络请求：OkHttp 拦截器。
- 应用日志：`AppLog`。
- 应用崩溃：`CrashHandler`。
- 书源/RSS 源调试：`Debug.log`。
- 规则解析：`AnalyzeRule`、`RuleAnalyzer`、`WebBook`、`Rss` 等关键阶段。
- 书源校验：`CheckSource`、`CheckSourceService`、`CheckSourceResultEvent`。

采集层原则：

- 不直接操作 UI。
- 不依赖 Activity 或 Fragment。
- 不做复杂过滤，只负责上报结构化事件。
- 不在网络拦截器里做重 IO。
- 对大字段做截断和脱敏。

## 调试事件模型

建议新增统一事件模型，例如 `DebugEvent`。

字段建议：

```text
DebugEvent
- id: String
- time: Long
- level: DebugLevel
- category: DebugCategory
- message: String
- detail: String?
- sourceName: String?
- sourceUrl: String?
- requestId: String?
- traceId: String?
- url: String?
- method: String?
- statusCode: Int?
- duration: Long?
- throwable: Throwable?
- tags: Map<String, String>
```

`DebugLevel` 建议：

```text
DEBUG
INFO
WARN
ERROR
```

`DebugCategory` 建议：

```text
APP
NETWORK
RULE
SOURCE
RSS
CHECK
CRASH
```

## traceId 设计

如果要真正判断“源规则执行正确与否”，只看单条日志不够。需要把一次操作里的网络请求、规则解析和错误串起来。

建议引入 `traceId`。

使用方式：

- 一次书源搜索生成一个 traceId。
- 搜索页请求、搜索规则解析、详情页请求、详情页规则解析都带同一个 traceId。
- 网络拦截器读取请求 Header 或上下文中的 traceId。
- UI 上可以按 traceId 展示一次完整链路。

例如：

```text
traceId = source-debug-1710000000000

1. 搜索页开始
2. GET https://example.com/search?q=xxx 200 320ms
3. 搜索规则执行成功，提取 12 条
4. 详情页开始
5. GET https://example.com/book/1 200 180ms
6. 详情规则失败：bookName 为空
```

## 调试事件中心

建议新增 `DebugEventCenter`。

职责：

- 接收所有调试事件。
- 提供实时事件流。
- 维护内存环形缓冲区。
- 根据配置决定是否持久化。
- 提供清空、导出、查询最近日志等能力。

内部可以使用：

```text
MutableSharedFlow<DebugEvent>
ArrayDeque<DebugEvent>
Mutex 或 synchronized
```

推荐策略：

- 内存保留最近 500 到 1000 条。
- 超过上限丢弃最旧事件。
- UI 订阅实时流。
- UI 打开时先读取内存快照，再订阅新增事件。

## 数据持久化

持久化不是第一版必须项。

第一版可以只做内存日志，避免数据库迁移和 IO 成本。

如果后续要落库，可以新增调试日志表，或复用现有 URL 记录表的一部分能力。

持久化建议：

- 只在用户开启调试记录时启用。
- 网络日志可以继续使用现有 URL 记录能力。
- 结构化调试事件可以单独建表。
- 大字段单独存储或直接不落库。
- 定期清理旧数据，避免数据库膨胀。

## 网络采集设计

现有 `UrlRecordInterceptor` 已经记录网络访问信息，可以扩展为同时上报 `DebugEvent`。

记录内容：

- URL
- 域名
- 方法
- 状态码
- 耗时
- 错误信息
- 来源书源名
- 来源书源 URL
- 请求体摘要

注意事项：

- 请求体只记录短文本。
- 响应体默认不记录。
- 响应体预览必须有开关。
- Cookie、Authorization、Token 等字段要脱敏。
- 拦截器里不要同步写数据库或做复杂格式化。

## 规则采集设计

现有 `Debug.callback` 偏向单页面调试，只适合一个调试界面消费。

建议演进为：

```text
Debug.log(...)
  ↓
DebugEventCenter.emit(...)
  ↓
现有 callback 继续兼容
```

也就是说，不直接废弃现有调试页面，而是在现有 `Debug.log` 里补充上报事件。

后续可以继续在关键阶段加入结构化事件：

- 搜索页开始
- 搜索页请求完成
- 搜索规则执行完成
- 详情页开始
- 详情规则执行完成
- 目录页开始
- 目录规则执行完成
- 正文页开始
- 正文规则执行完成
- 阶段失败

这样 UI 可以从文本日志升级为链路化展示。

## 错误采集设计

错误来源包括：

- `AppLog.put`
- `AppLog.putDebug`
- `CrashHandler`
- OkHttp 请求异常
- 规则解析异常
- 协程任务失败

错误事件建议记录：

- 错误摘要
- 异常类型
- 堆栈
- 所属书源
- 所属 URL
- traceId
- 所属阶段

UI 上错误日志应支持快速复制详情。

## 悬浮球实现方案

推荐使用应用内悬浮球。

实现方式：

- 通过 `BaseActivity` 或 `Application.ActivityLifecycleCallbacks` 挂载。
- 将悬浮 View 添加到当前 Activity 的 `decorView`。
- Activity 切换时重新挂载。
- 支持拖动位置记忆。
- 支持页面白名单或黑名单。

优点：

- 不需要系统悬浮窗权限。
- 生命周期可控。
- 不涉及跨 App 显示。
- 更适合应用内调试需求。

暂不推荐系统悬浮窗：

- 需要额外权限。
- Android 版本兼容复杂。
- 容易被系统限制。
- 对当前需求收益不高。

## 调试 UI 设计

调试 UI 可以是 Activity、DialogFragment 或 BottomSheet。

建议第一版使用独立 Activity 或全屏 Dialog：

- 展示空间更充足。
- 日志详情更容易阅读。
- 不容易遮挡原页面。

页面结构：

```text
顶部工具栏
  暂停/继续
  清空
  导出
  过滤

分类 Tab
  全部
  网络
  规则
  错误
  校验

日志列表
  时间
  类型
  摘要
  状态
  耗时

详情页
  完整 URL
  请求信息
  来源书源
  规则阶段
  异常堆栈
```

## 性能策略

需要避免调试功能影响主流程。

建议：

- 默认只保留内存最近日志。
- 大字段截断。
- UI 列表分页或虚拟化。
- 高频日志合并或限流。
- 拦截器只上报轻量事件。
- 数据库写入放到 IO 协程。
- 悬浮球关闭面板后仍可采集，但不做 UI 渲染。

## 隐私与安全

调试日志可能包含敏感信息。

必须考虑：

- Cookie 脱敏。
- Authorization 脱敏。
- Token、key、password 等字段脱敏。
- 请求体和响应体默认不完整记录。
- 导出日志前可以提示敏感信息风险。
- Release 版本默认关闭高级采集。

## 配置项

建议增加调试配置：

- 是否启用悬浮球。
- 是否记录网络日志。
- 是否记录规则日志。
- 是否记录错误日志。
- 是否记录请求体。
- 是否记录响应体。
- 最大内存日志数量。
- 是否允许导出日志。

这些配置可以挂在现有设置页或调试工具页中。

## MVP 落地步骤

第一阶段：

1. 新增统一 `DebugEvent` 模型。
2. 新增 `DebugEventCenter`。
3. 在 `AppLog` 上报应用日志事件。
4. 在 `UrlRecordInterceptor` 上报网络事件。
5. 在 `Debug.log` 上报规则调试事件。
6. 新增应用内悬浮球入口。
7. 新增实时日志页面。
8. 支持过滤、暂停、清空、复制详情。

第二阶段：

1. 增加 traceId。
2. 增加规则执行阶段事件。
3. 增加书源校验进度聚合。
4. 增加导出日志。
5. 增加响应体预览。
6. 增加敏感字段脱敏配置。

## 风险点

主要风险：

- 日志太多导致 UI 卡顿。
- 网络响应体采集影响性能或破坏流读取。
- 现有 `Debug.callback` 只能单订阅，扩展时要兼容。
- 日志里包含隐私信息。
- 悬浮球遮挡业务 UI。
- Activity 切换时悬浮球重复挂载或泄漏。

规避方式：

- 使用内存环形缓冲区。
- 默认不采集响应体。
- 保留现有 callback，同时新增事件中心。
- 对敏感字段脱敏。
- 支持拖动、隐藏、页面黑名单。
- 统一由 Activity 生命周期管理悬浮球。

## 结论

悬浮日志球的架构核心是统一调试事件系统。

推荐先完成轻量 MVP：应用内悬浮球、实时日志页面、应用日志、网络日志、现有规则调试日志接入。等基础链路稳定后，再做 traceId、结构化规则阶段、日志导出和响应体预览。

这样既能快速落地，又不会把后续调试能力锁死在一个简单弹窗里。
