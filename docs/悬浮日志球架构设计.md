# 悬浮日志球架构设计

第二份偏工程架构，包含事件模型、采集层、事件中心、traceId、悬浮球实现、UI、性能和风险点

## 设计目标

构建一个应用内实时调试控制台，通过页面右下角悬浮球进入。系统需要统一采集网络请求、应用错误、书源/RSS 源规则执行、书源校验等事件，并以低侵入、可扩展的方式展示给用户。

设计重点：

- 采集和展示解耦。
- 日志事件结构化。
- 实时展示，不依赖 UI 轮询。
- 默认低开销，避免影响阅读和网络性能。
- 对请求体、响应体、Cookie、Token 等敏感内容做保护。
- 能复用现有 `AppLog`、`UrlRecordInterceptor`、`Debug`、书源调试和校验能力。

## 总体分层

建议分为四层：

```text
采集层
  ↓
调试事件中心
  ↓
调试数据仓库
  ↓
悬浮入口与调试 UI
```

## 采集层

采集层负责把不同模块里的运行信息转换成统一的调试事件。

主要采集点：

- 网络请求：OkHttp 拦截器。
- 应用日志：`AppLog`。
- 应用崩溃：`CrashHandler`。
- 书源/RSS 源调试：`Debug.log`。
- 规则解析：`AnalyzeRule`、`RuleAnalyzer`、`WebBook`、`Rss` 等关键阶段。
- 书源校验：`CheckSource`、`CheckSourceService`、`CheckSourceResultEvent`。

采集层原则：

- 不直接操作 UI。
- 不依赖 Activity 或 Fragment。
- 不做复杂过滤，只负责上报结构化事件。
- 不在网络拦截器里做重 IO。
- 对大字段做截断和脱敏。

## 调试事件模型

建议新增统一事件模型，例如 `DebugEvent`。

字段建议：

```text
DebugEvent
- id: String
- time: Long
- level: DebugLevel
- category: DebugCategory
- message: String
- detail: String?
- sourceName: String?
- sourceUrl: String?
- requestId: String?
- traceId: String?
- url: String?
- method: String?
- statusCode: Int?
- duration: Long?
- throwable: Throwable?
- tags: Map<String, String>
```

`DebugLevel` 建议：

```text
DEBUG
INFO
WARN
ERROR
```

`DebugCategory` 建议：

```text
APP
NETWORK
RULE
SOURCE
RSS
CHECK
CRASH
```

## traceId 设计

如果要真正判断“源规则执行正确与否”，只看单条日志不够。需要把一次操作里的网络请求、规则解析和错误串起来。

建议引入 `traceId`。

使用方式：

- 一次书源搜索生成一个 traceId。
- 搜索页请求、搜索规则解析、详情页请求、详情页规则解析都带同一个 traceId。
- 网络拦截器读取请求 Header 或上下文中的 traceId。
- UI 上可以按 traceId 展示一次完整链路。

例如：

```text
traceId = source-debug-1710000000000

1. 搜索页开始
2. GET https://example.com/search?q=xxx 200 320ms
3. 搜索规则执行成功，提取 12 条
4. 详情页开始
5. GET https://example.com/book/1 200 180ms
6. 详情规则失败：bookName 为空
```

## 调试事件中心

建议新增 `DebugEventCenter`。

职责：

- 接收所有调试事件。
- 提供实时事件流。
- 维护内存环形缓冲区。
- 根据配置决定是否持久化。
- 提供清空、导出、查询最近日志等能力。

内部可以使用：

```text
MutableSharedFlow<DebugEvent>
ArrayDeque<DebugEvent>
Mutex 或 synchronized
```

推荐策略：

- 内存保留最近 500 到 1000 条。
- 超过上限丢弃最旧事件。
- UI 订阅实时流。
- UI 打开时先读取内存快照，再订阅新增事件。

## 数据持久化

持久化不是第一版必须项。

第一版可以只做内存日志，避免数据库迁移和 IO 成本。

如果后续要落库，可以新增调试日志表，或复用现有 URL 记录表的一部分能力。

持久化建议：

- 只在用户开启调试记录时启用。
- 网络日志可以继续使用现有 URL 记录能力。
- 结构化调试事件可以单独建表。
- 大字段单独存储或直接不落库。
- 定期清理旧数据，避免数据库膨胀。

## 网络采集设计

现有 `UrlRecordInterceptor` 已经记录网络访问信息，可以扩展为同时上报 `DebugEvent`。

记录内容：

- URL
- 域名
- 方法
- 状态码
- 耗时
- 错误信息
- 来源书源名
- 来源书源 URL
- 请求体摘要

注意事项：

- 请求体只记录短文本。
- 响应体默认不记录。
- 响应体预览必须有开关。
- Cookie、Authorization、Token 等字段要脱敏。
- 拦截器里不要同步写数据库或做复杂格式化。

## 规则采集设计

现有 `Debug.callback` 偏向单页面调试，只适合一个调试界面消费。

建议演进为：

```text
Debug.log(...)
  ↓
DebugEventCenter.emit(...)
  ↓
现有 callback 继续兼容
```

也就是说，不直接废弃现有调试页面，而是在现有 `Debug.log` 里补充上报事件。

后续可以继续在关键阶段加入结构化事件：

- 搜索页开始
- 搜索页请求完成
- 搜索规则执行完成
- 详情页开始
- 详情规则执行完成
- 目录页开始
- 目录规则执行完成
- 正文页开始
- 正文规则执行完成
- 阶段失败

这样 UI 可以从文本日志升级为链路化展示。

## 错误采集设计

错误来源包括：

- `AppLog.put`
- `AppLog.putDebug`
- `CrashHandler`
- OkHttp 请求异常
- 规则解析异常
- 协程任务失败

错误事件建议记录：

- 错误摘要
- 异常类型
- 堆栈
- 所属书源
- 所属 URL
- traceId
- 所属阶段

UI 上错误日志应支持快速复制详情。

## 悬浮球实现方案（Compose 现代化）

推荐使用 **Jetpack Compose** 实现应用内悬浮球。

### 技术选型

**方案一：Activity 内 Compose 悬浮球（推荐）**

```kotlin
@Composable
fun DebugFloatingBall(
    onClick: () -> Unit,
    modifier: Modifier = Modifier,
    isVisible: Boolean = true
) {
    if (!isVisible) return

    var offset by remember { mutableStateOf(Offset(x = 0f, y = 0f)) }
    val configuration = LocalConfiguration.current

    Box(
        modifier = modifier
            .offset { IntOffset(offset.x.roundToInt(), offset.y.roundToInt()) }
            .size(56.dp)
            .shadow(elevation = 8.dp, shape = CircleShape)
            .clip(CircleShape)
            .background(MaterialTheme.colorScheme.primary)
            .clickable(
                interactionSource = remember { MutableInteractionSource() },
                indication = ripple(bounded = false, radius = 28.dp),
                onClick = onClick
            )
            .pointerInput(Unit) {
                detectDragGestures(
                    onDragStart = { /* 可记录起始位置 */ },
                    onDragEnd = {
                        // 吸附到屏幕边缘
                        val screenWidth = configuration.screenWidthDp.dp.toPx()
                        offset = if (offset.x > screenWidth / 2) {
                            Offset(screenWidth - 56.dp.toPx(), offset.y)
                        } else {
                            Offset(0f, offset.y)
                        }
                    },
                    onDragCancel = { /* 取消拖拽 */ },
                    onDragChange = { change, dragAmount ->
                        change.consume()
                        offset += dragAmount
                    }
                }),
        contentAlignment = Alignment.Center
    ) {
        Icon(
            imageVector = Icons.Default.BugReport,
            contentDescription = "调试日志",
            tint = Color.White,
            modifier = Modifier.size(28.dp)
        )

        // 未读日志数量徽章
        if (unreadCount > 0) {
            Box(
                modifier = Modifier
                    .align(Alignment.TopEnd)
                    .offset(x = (-4).dp, y = 4.dp)
                    .size(20.dp)
                    .background(Color.Red, CircleShape),
                contentAlignment = Alignment.Center
            ) {
                Text(
                    text = unreadCount.coerceAtMost(99).toString(),
                    color = Color.White,
                    style = MaterialTheme.typography.labelSmall
                )
            }
        }
    }
}
```

**方案二：跨 Activity 悬浮球（WindowManager + ComposeView）**

```kotlin
class DebugFloatingBallManager(private val context: Context) {

    private var windowManager: WindowManager? = null
    private var composeView: ComposeView? = null

    fun show() {
        windowManager = context.getSystemService(Context.WINDOW_SERVICE) as WindowManager

        composeView = ComposeView(context).apply {
            setContent {
                AppTheme {
                    DebugFloatingBall(onClick = ::openDebugPanel)
                }
            }
        }

        val params = WindowManager.LayoutParams(
            WindowManager.LayoutParams.WRAP_CONTENT,
            WindowManager.LayoutParams.WRAP_CONTENT,
            WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY,
            WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE or
                    WindowManager.LayoutParams.FLAG_LAYOUT_NO_LIMITS,
            PixelFormat.TRANSLUCENT
        ).apply {
            gravity = Gravity.TOP or Gravity.START
            x = 0
            y = 500
        }

        windowManager?.addView(composeView, params)
    }

    fun dismiss() {
        composeView?.let { windowManager?.removeView(it) }
    }
}
```

### 核心优势

- **声明式 UI**：状态变化自动触发重组，无需手动更新 View。
- **手势处理现代化**：使用 `pointerInput` + `detectDragGestries` 替代复杂的 TouchListener。
- **生命周期感知**：`remember` 自动管理状态，避免内存泄漏。
- **动画内置支持**：可轻松集成 Compose Animation API（如吸附动画）。
- **主题一致性**：直接复用 Material3 主题系统。

```kotlin
@Composable
fun rememberFloatingBallPosition(): State<Offset> {
    return produceState(initialValue = Offset(100f, 800f)) {
        context.dataStore.data.collect { prefs ->
            value = Offset(
                x = prefs[floatPreferencesKey("ball_x")] ?: 100f,
                y = prefs[floatPreferencesKey("ball_y")] ?: 800f

        }
    }
```

### 页面过滤策略

```kotlin
@Composable
fun DebugFloatingBallWithFilter(
    currentRoute: String?,
    blackListRoutes: Set<String> = setOf("/splash", "/webview"),

) {
    // 在黑名单页面不显示悬浮球
    val shouldShow = currentRoute !in blackListRoutes
```

**暂不推荐传统 View 方案：**

- 手动管理 View 生命周期复杂。
- 需要处理 Activity 切换时的重复挂载问题。
- 触摸事件处理代码冗长且难以维护。

```kotlin
        }
```

```kotlin
@Composable
private fun DebugLogItem(
    log: DebugEvent,
    onClick: (DebugEvent) -> Unit,
    modifier: Modifier = Modifier
) {
    Surface(
        modifier = modifier
            .fillMaxWidth()
            .clickable(onClick = { onClick(log) }),
        shape = RoundedCornerShape(12.dp),
        color = when (log.level) {
            DebugLevel.ERROR -> MaterialTheme.colorScheme.errorContainer
            DebugLevel.WARN -> MaterialTheme.colorScheme.tertiaryContainer
            else -> MaterialTheme.colorScheme.surfaceVariant
        },
        tonalElevation = 2.dp
    ) {
        Row(
            modifier = Modifier.padding(12.dp),
            verticalAlignment = Alignment.CenterVertically
        ) {
            // 级别图标
            Icon(
                imageVector = when (log.level) {
                    DebugLevel.ERROR -> Icons.Default.ErrorOutline
                    DebugLevel.WARN -> Icons.Default.WarningAmber
                    else -> Icons.Default.InfoOutline
                },
                contentDescription = log.level.name,
                tint = when (log.level) {
                    DebugLevel.ERROR -> MaterialTheme.colorScheme.error
                    DebugLevel.WARN -> MaterialTheme.colorScheme.tertiary
                    else -> MaterialTheme.colorScheme.primary
                },
                modifier = Modifier.size(24.dp)
            )

            Spacer(Modifier.width(12.dp))

            // 内容区域
            Column(modifier = Modifier.weight(1f)) {
                Text(
                    text = log.message,
                    style = MaterialTheme.typography.bodyMedium,
                    maxLines = 2,
                    overflow = TextOverflow.Ellipsis,
                    color = when (log.level) {
                        DebugLevel.ERROR -> MaterialTheme.colorScheme.onErrorContainer
                        DebugLevel.WARN -> MaterialTheme.colorScheme.onTertiaryContainer
                        else -> MaterialTheme.colorScheme.onSurfaceVariant
                    }
                )

                Spacer(Modifier.height(4.dp))

                Row(
                    horizontalArrangement = Arrangement.spacedBy(8.dp)
                ) {
                    // 时间戳
                    Text(
                        text = formatRelativeTime(log.time),
                        style = MaterialTheme.typography.labelSmall,
                        color = MaterialTheme.colorScheme.outline
                    )

                    // 分类标签
                    if (log.category != DebugCategory.APP) {
                        Surface(
                            shape = RoundedCornerShape(4.dp),
                            color = MaterialTheme.colorScheme.secondaryContainer.copy(alpha = 0.5f)
                        ) {
                            Text(
                                text = log.category.name,
                                style = MaterialTheme.typography.labelSmall,
                                color = MaterialTheme.colorScheme.onSecondaryContainer,
                                modifier = Modifier.padding(horizontal = 6.dp, vertical = 2.dp)
                            )
                        }
                    }
                }
            }

            // 耗时标签（可选）
            if (log.duration != null && log.duration > 0) {
                Text(
                    text = "${log.duration}ms",
                    style = MaterialTheme.typography.labelSmall,
                    color = if (log.duration > 1000)
                        MaterialTheme.colorScheme.error
                    else
                        MaterialTheme.colorScheme.outline,
                    modifier = Modifier.padding(start = 8.dp)
                )
            }
        }
    }
}
```

## 性能策略（Compose 专项优化）

需要避免调试功能影响主流程。

### 基础性能策略

- 默认只保留内存最近日志。
- 大字段截断。
- 拦截器只上报轻量事件。
- 数据库写入放到 IO 协程。
- 悬浮球关闭面板后仍可采集，但不做 UI 渲染。

#### 7. 内存管理

```kotlin
class DebugEventCenter {

    private val _events = ArrayDeque<DebugEvent>()
    private val mutex = Mutex()

    companion object {
        const val MAX_EVENTS = 500  // 内存中最多保留500条
    }

    suspend fun emit(event: DebugEvent) {
        withContext(Dispatchers.IO) {
            mutex.withLock {
                _events.addFirst(event)

                // 超过上限时移除最旧的
                while (_events.size > MAX_EVENTS) {
                    _events.removeLast()
                }
            }
        }
    }

    fun getRecentLogs(limit: Int): List<DebugEvent> {
        return synchronized(_events) {
            _events.take(limit)
        }
    }
}
```

## 隐私与安全

调试日志可能包含敏感信息。

必须考虑：

- Cookie 脱敏。
- Authorization 脱敏。
- Token、key、password 等字段脱敏。
- 请求体和响应体默认不完整记录。
- 导出日志前可以提示敏感信息风险。
- Release 版本默认关闭高级采集。

## 配置项

建议增加调试配置：

- 是否启用悬浮球。
- 是否记录网络日志。
- 是否记录规则日志。
- 是否记录错误日志。
- 是否记录请求体。
- 是否记录响应体。
- 最大内存日志数量。
- 是否允许导出日志。

这些配置可以挂在现有设置页或调试工具页中。

```kotlin
class UrlRecordInterceptor : Interceptor {

    override fun intercept(chain: Interceptor.Chain): Response {
        val startTime = System.currentTimeMillis()
        val request = chain.request()
        var response: Response? = null
        var exception: Throwable? = null

        try {
            response = chain.proceed(request)
            response
        } catch (e: Exception) {
            exception = e
            throw e
        } finally {
            // 上报网络事件
            viewModelScope.launch(Dispatchers.IO) {
                DebugEventCenter.emit(
                    DebugEvent(
                        level = if (exception != null || response?.isSuccessful != true)
                            DebugLevel.ERROR else DebugLevel.INFO,
                        category = DebugCategory.NETWORK,
                        message = "${request.method} ${SensitiveDataUtils.sanitizeUrl(request.url.toString())}",
                        url = SensitiveDataUtils.sanitizeUrl(request.url.toString()),
                        method = request.method,
                        statusCode = response?.code,
                        duration = System.currentTimeMillis() - startTime,
                        throwable = exception
                    )
                )
            }
        }
    }
}
```

#### 1. 日志太多导致 UI 卡顿（重组风暴）

**风险描述：**
高频事件流导致 Compose 频繁重组，造成 UI 卡顿甚至 ANR。

**规避方式：**

```kotlin

    extraBufferCapacity = 64,
    onBufferOverflow = BufferOverflow.DROP_OLDEST
)

init {
    viewModelScope.launch {
        _eventFlow
            .debounce(100)  // 100ms 内只处理最新事件
            .collectLatest { event ->
                // 使用 snapshotStateList 或 diffUtil 更新列表
                updateLogs(event)
            }
    }
}

// ✅ 在 LazyColumn 中使用稳定的 key

    DebugLogItem(log)
```

#### 2. 网络响应体采集影响性能或破坏流读取

**风险描述：**
OkHttp 响应体只能读取一次，采集响应体会导致业务代码无法再次读取。

**规避方式：**

```kotlin
class NetworkLoggingInterceptor : Interceptor {

    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request()
        val response = chain.proceed(request)

        // ✅ 只在配置开启时采集，且使用 .peek() 不消费 body
        if (shouldLogResponse(request)) {
            val responseBody = response.body
            if (responseBody != null) {
                val source = responseBody.source()
                source.request(Long.MAX_VALUE)
                val buffer = source.buffer.clone()

                // 限制预览长度
                val preview = if (buffer.size > MAX_PREVIEW_SIZE) {
                    buffer.readByteString(MAX_PREVIEW_SIZE).utf8() + "...(truncated)"
                } else {
                    buffer.readUtf8()
                }

                emitNetworkEvent(
                    url = request.url.toString(),
                    responsePreview = preview,
                    statusCode = response.code,
                    duration = System.currentTimeMillis() - startTime
                )
            }
        }

        return response
    }

    companion object {
        const val MAX_PREVIEW_SIZE = 1024 * 4  // 最大 4KB
    }
}
```

**规避方式：**

```kotlin
object DebugEventCenter {
    private val _eventFlow = MutableSharedFlow<DebugEvent>(
        replay = 0,  // 不重放历史给新订阅者（避免重复）
        extraBufferCapacity = 256,
        onBufferOverflow = BufferOverflow.DROP_OLDEST
    )

    val eventFlow: SharedFlow<DebugEvent> = _eventFlow.asSharedFlow()

    suspend fun emit(event: DebugEvent) {
        _eventFlow.emit(event)
    }

    // 兼容旧 callback
    fun setLegacyCallback(callback: (String) -> Unit) {
        viewModelScope.launch {
            eventFlow.collect { event ->
                callback(event.toLegacyFormat())
            }
        }
    }
}
```

#### 4. 日志里包含隐私信息

**风险描述：**
Cookie、Token、密码等敏感信息可能被记录到日志中。

**规避方式：**

```kotlin
object SensitiveDataUtils {

    private val SENSITIVE_HEADERS = setOf(
        "cookie", "authorization", "x-auth-token",
        "api-key", "password", "token"
    )

    private val SENSITIVE_PARAMS = setOf(
        "token", "key", "password", "secret",
        "credential", "access_token"
    )

    fun sanitizeUrl(url: String): String {
        return try {
            val uri = Uri.parse(url)
            uri.buildUpon().apply {
                // 清除敏感参数
                uri.queryParameterNames.forEach { param ->
                    if (param.lowercase() in SENSITIVE_PARAMS) {
                        appendQueryParameter(param, "***")
                    }
                }
            }.build().toString()
        } catch (e: Exception) {
            url
        }
    }

    fun sanitizeHeaders(headers: Map<String, String>): Map<String, String> {
        return headers.mapValues { (key, value) ->
            if (key.lowercase() in SENSITIVE_HEADERS) "***" else value
        }
    }

    fun sanitizeBody(body: String?): String? {
        if (body.isNullOrBlank()) return body

        var sanitized = body
        SENSITIVE_PARAMS.forEach { keyword ->
            sanitized = sanitized.replace(
                Regex("(?i)\"($keyword)\"\\s*:\\s*\"[^\"]*\"", RegexOption.IGNORE_CASE),
                "\"$1\":\"***\""
            )
        }
        return sanitized
    }
}
```

```kotlin
class GlobalDebugFloatingBallManager(private val context: Context) {

    private var windowManager: WindowManager? = null
    private var composeView: ComposeView? = null
    private val _isVisible = MutableStateFlow(true)
    val isVisible: StateFlow<Boolean> = _isVisible.asStateFlow()

    fun show() {
        if (composeView != null) return  // 避免重复挂载

        windowManager = context.getSystemService(Context.WINDOW_SERVICE) as WindowManager

        composeView = ComposeView(context).apply {
            setContent {
                val visible by _isVisible.collectAsState()
                if (visible) {
                    DebugFloatingBall(onClick = ::openDebugPanel)
                }
            }
        }

        val params = WindowManager.LayoutParams(
            WindowManager.LayoutParams.WRAP_CONTENT,
            WindowManager.LayoutParams.WRAP_CONTENT,
            WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY,
            WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE or
                    WindowManager.LayoutParams.FLAG_LAYOUT_NO_LIMITS,
            PixelFormat.TRANSLUCENT
        ).apply {
            gravity = Gravity.TOP or Gravity.START
            x = loadSavedPosition().first
            y = loadSavedPosition().second
        }

        try {
            windowManager?.addView(composeView, params)
        } catch (e: Exception) {
            Log.e("DebugBall", "Failed to add floating ball", e)
        }

    fun dismiss() {
        composeView?.let {
            try {
                windowManager?.removeView(it)
            } catch (e: Exception) {
                Log.e("DebugBall", "Failed to remove floating ball", e)
            } finally {
                composeView = null
            }
        }
    }

    fun toggleVisibility() {
        _isVisible.value = !_isVisible.value
    }
}
```

```kotlin
        val duration = (System.nanoTime() - startTime) / 1_000_000  // ms

        if (duration > 16) {  // 超过一帧时间（60fps）
            Log.w("Performance", "$tag composition took ${duration}ms (>16ms)")
        }
    }
}
```

## 结论

悬浮日志球的架构核心是统一调试事件系统。

推荐先完成轻量 MVP：应用内悬浮球、实时日志页面、应用日志、网络日志、现有规则调试日志接入。等基础链路稳定后，再做 traceId、结构化规则阶段、日志导出和响应体预览。

这样既能快速落地，又不会把后续调试能力锁死在一个简单弹窗里。
