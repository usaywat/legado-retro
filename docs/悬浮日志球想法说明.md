# 悬浮日志球想法说明

第一份偏想法、产品定位和 MVP；

## 背景

希望在软件页面右下角常驻一个悬浮球。用户点开后，可以进入一个实时日志页面，用来观察当前应用运行状态，尤其是网络请求、报错信息、书源/RSS 源规则执行过程和校验结果。

这个功能不应该只是把日志文本展示出来，而应该成为一个应用内调试控制台。它可以把现有的日志、网络记录、书源调试、RSS 调试、校验结果等能力整合到一个统一入口里。

## 核心目标

- 页面右下角常驻悬浮球，方便随时打开调试面板。
- 实时显示网络请求日志，包括 URL、请求方法、状态码、耗时、错误信息、来源书源等。
- 实时显示应用错误，包括普通错误、异常堆栈、崩溃日志摘要。
- 实时显示书源和 RSS 源规则执行状态。
- 能看出规则执行是否正确，例如搜索页、详情页、目录页、正文页分别成功或失败。
- 支持过滤、暂停、清空、复制、导出等基础调试操作。

## 功能定位

悬浮球只是入口，真正重要的是背后的统一调试事件管道。

如果只做一个悬浮按钮打开日志列表，后续会很难扩展。更合理的方式是先把不同来源的调试信息统一成一种事件模型，再由悬浮日志页面消费这些事件。

建议定位为：

> 应用内实时调试控制台。

它可以覆盖：

- 网络请求观察
- 书源规则调试
- RSS 源规则调试
- 书源校验过程
- 应用错误和崩溃摘要
- 开发调试信息

## 推荐交互

悬浮球默认显示在页面右下角。

行为建议：

- 可拖动，避免遮挡阅读、列表、按钮等区域。
- 可点击展开调试面板。
- 可长按隐藏或进入设置。
- 有错误时可显示红点或错误数量。
- 有网络活动时可显示轻微状态变化。
- 面板关闭后继续采集日志。

调试面板建议包含：

- 顶部工具栏：暂停/继续、清空、导出、过滤。
- 分类标签：全部、网络、规则、错误、校验。
- 日志列表：按时间倒序或顺序显示。
- 详情页面：点击日志项查看完整信息。

## 日志分类

建议至少分为以下几类：

- `全部`：显示所有调试事件。
- `网络`：显示 HTTP 请求、响应、异常、耗时。
- `规则`：显示书源/RSS 源规则解析过程。
- `错误`：显示异常、解析失败、请求失败、崩溃摘要。
- `校验`：显示书源校验进度和结果。

## 网络日志内容

第一版建议记录：

- 请求时间
- 请求 URL
- 请求方法
- 状态码
- 耗时
- 域名
- 来源书源名称
- 来源书源 URL
- 错误信息

可选记录：

- 请求头
- 请求体
- 响应头
- 响应体摘要

注意：请求体和响应体不建议默认完整记录，应该加开关和长度限制。

## 规则执行日志内容

书源/RSS 源调试不应该只显示“成功”或“失败”，而应该按阶段显示。

例如书源调试可以拆成：

- 搜索页开始
- 搜索页请求完成
- 搜索规则执行完成，提取到多少条结果
- 详情页开始
- 详情页规则执行完成或失败
- 目录页开始
- 目录页规则执行完成或失败
- 正文页开始
- 正文页规则执行完成，正文长度多少

这样排查问题时可以直接看到失败发生在哪一步。

## 第一版 MVP

建议第一版不要做太重，先完成可用闭环：

1. 应用内右下角悬浮球。
2. 点开显示实时日志面板。
3. 接入应用日志。
4. 接入网络请求日志。
5. 接入现有书源/RSS 调试日志。
6. 支持全部、网络、规则、错误过滤。
7. 支持暂停滚动、清空日志、复制详情。

## 第二版增强

后续可以继续扩展：

- 结构化规则执行追踪。
- 网络请求和规则执行通过 traceId 串联。
- 导出调试日志文件。
- 响应体预览。
- 敏感字段自动脱敏。
- 书源校验实时进度聚合。
- 错误聚合和统计。
- 根据书源名称或域名过滤。

## 不建议的方向

不建议直接把 Logcat 搬进 UI。Logcat 噪声太大，而且和书源规则、网络请求没有明确结构关系。

不建议 UI 直接轮询数据库。实时性差，也容易增加 IO 压力。

不建议第一版做系统级悬浮窗。系统悬浮窗需要额外权限，兼容成本更高。除非明确需要跨 App 显示，否则应用内悬浮球更合适。

## 总结

这个功能的关键不是悬浮球本身，而是统一的实时调试事件系统。悬浮球只是让用户随时进入调试控制台。

只要事件管道设计好，后续无论是 App 内调试面板、WebSocket 调试、日志导出、书源校验报告，都可以复用同一套数据。
